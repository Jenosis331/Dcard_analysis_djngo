### Phase 2：進階輿情分析代理人 (基於 LangGraph 與 Gemini 1.5/2.0)

#### 1. 使用到的套件
在第二階段的實作中，我們升級了整體的 AI 架構，主要依賴以下幾個關鍵套件：
*   **`langgraph` 與 `langgraph-prebuilt`**：這是 LangChain 官方最新的狀態機框架。取代了舊有的 AgentExecutor，用來打造更能掌控決策與執行迴圈的 AI Agent（代理人）。
*   **`langchain-google-genai`**：強大的橋樑套件，讓我們能直接串接 Google 的 `gemini-1.5-flash` 或 `gemini-2.0-flash` 語言模型。
*   **`langchain-core`**：負責定義工具 (Tools)、格式化提示詞 (Prompt) 以及訊息結構 (Messages) 的核心基礎庫。
*   **`Django` (Session)**：藉由 Django 的 Session 狀態管理（如 `views.py` 中實作），負責持久化對話紀錄 `chat_history`。這賦予了 AI 持續性的上下文記憶，並支援「清除記憶」功能以開啟新對話。

#### 2. 想要達到的目的
第一階段雖然導入了 AI，但 AI 單純只是無狀態（Stateless）的「一問一答」客服；第二階段的核心目的是要打造一位**「具備推理與自主執行力的資料分析師 (Agentic Workflow)」**。我們將資料庫查詢邏輯封裝為 AI 專屬工具，讓 Agent 能根據使用者意圖判斷查詢順序，並將多源數據整合為一份具備深度洞察的輿情報告。

#### 3. 程式具備的功能
*   **自動化工具調用 (Tool Calling)**：程式內部註冊了查詢趨勢 (`get_top_keywords`)、人物掃描 (`get_top_persons`)、算關聯度 (`get_keyword_correlation`) 與段落擷取 (`get_keyword_paragraphs`) 等功能。AI 具備「選擇工具」的能力。
*   **記憶維持機制**：透過 `views.py` 將歷史訊息與新問題一併傳入 Agent，實現連貫的對話能力。使用者可以用「那『他』最近的新聞呢？」這種代名詞追問，AI 會自動從歷史上下文中解析主體。
*   **動態資訊融合與清理**：無論工具回傳的是 JSON 數據或原始文本段落，Agent 都能精準萃取關鍵資訊，並透過模型強大的語意理解進行「情緒標註與多維總結」，最後以結構化的 Markdown 格式輸出。

#### 4. 整體運作流程
1.  **前台發問與記憶檢索**：使用者發問後，`views.py` 從 Session 提取 `chat_history` 並傳入 `run_agent_phase2`。
2.  **派發給 LangGraph 核心大腦**：被打包的訊息進入了 `create_react_agent` 打造的決策控制中心。
3.  **大腦決策與迭代迴圈 (ReAct 機制)**：
    *   大腦首先思考 (Thought)：這問題需要客觀資料嗎？
    *   若需要，則暫停回答並採取行動 (Action)：自動向我們的 Python 工具請求資料。
    *   觀察回傳結果 (Observation)：拿回資料後，判斷資訊足不足夠。若不足，它甚至能反覆發動二次、三次查詢（例如查完關聯度，接著加查新聞段落）。
4.  **生成洞察報表與狀態同步**：Agent 將生硬數據整合成結構化報告。`views.py` 接收後，將包含最新對話的歷史紀錄寫回 Session，並標記 `request.session.modified = True` 以確保狀態更新。

#### 5. 舉一個實際例子來展示整個流程
**【情境】使用者輸入：「分析『台積電』與『張忠謀』的關聯度，並列出最近相關的新聞段落。」**

**【Agent 的運作實境】**
1.  **AI 接收指令**：判斷問題涉及具體數據，決定啟動工具調用流程。
2.  **第一次 Action**：背景自動觸發 `get_keyword_correlation(query_a="台積電", query_b="張忠謀")`，獲取客觀相關係數（如 0.85）。
3.  **AI 繼續思考**：用戶還要求看新聞段落，所以任務還沒結束。
4.  **第二次 Action**：隨即觸發 `get_keyword_paragraphs(keyword="張忠謀 台積電")` 撈取原始新聞段落。
5.  **生成最終回答**：Agent 將數據、文本與語意分析進行深度融合，生成一篇結構完整且具備專業洞察的 Markdown 報告。



這是一個非常深入且關鍵的問題！答案是：**「不需要在 System Prompt 裡面手動列出所有函數的目的，他自己其實也能靈活運用！」**

讓我為您詳細說明背後的原因與運作機制：

### 為什麼不需要把每個 Tool 都寫在 System Prompt 裡？

在 LangChain / LangGraph 或是原生 Gemini 裡面，當我們把 `tools = [get_top_keywords, get_top_persons, ...]` 丟給模型時，底層的 SDK 會自動做一件事情：
**「將這些函數的 Python Docstring（註解）、參數型別（Type Hint），自動轉換成符合模型規範的 JSON Schema (Function Calling 定義檔) 傳送給模型。」**

所以即使我們在 System Prompt 裡面隻字未提，模型依然能夠從下面這些地方清楚知道這個工具的用途：
1. **函數名稱本身**（例如叫 `get_top_keywords`）。
2. **函數裡面的 Docstring**（例如 `"""取得過去幾週熱門新聞的關鍵字"""`）。
3. **參數型別與說明**（Type Hints）。

### 2. 在 System Prompt 加入說明的戰略意義 (為什麼「建議要寫」)

雖然模型知道工具的存在，但在 System Prompt 中進行點名與引導，是為了實現 **「戰術指導 (Steering)」**：

1. **教導更複雜的使用情境：** 
   例如教導 Agent：*「若要判斷關鍵字的情緒，請先呼叫 get_keyword_paragraphs 提取原始段落，再由你發揮 LLM 的語意分析能力進行判斷」*。這種「工具組合技」是 Docstring 無法表達的。
2. **避免幻覺與優先順序：**
   當系統工具有 20 個以上時，模型有時候會搞混。明確寫出哪種情境「一定」要呼叫哪個工具，可以確保它走在我們預期的康莊大道上，不會胡亂猜測。

### 結論與實務建議：

* **如果工具少於 10 個，且每個工具の功能都很明確獨立**：
  你完全可以把 `system_prompt` 砍到只剩一句話：`"你是專業的輿情分析專家（POA Agent V2）。你可以使用提供的工具來輔助你的回答。"`。剩下的全部交給模型憑藉著 Python Docstring 自動判斷。
* **如果工具數量很龐大 (例如 50 個)**：
  不建議在 Prompt 列出清單（會浪費 Token 並造成干擾），應專注於撰寫高品質的 Docstring 並配合 RAG 等技術進行動態工具檢索。

總結來說：**Docstring 是工具的「規格書」，而 System Prompt 則是 Agent 的「戰術手冊」。**
